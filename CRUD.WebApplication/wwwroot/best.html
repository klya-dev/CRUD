Лучшие практики

Настройка эндпоинта
  • REST API naming
  • MapGroup
  • HTTP метод (GET, POST)
  • Политики (.RequireAuthorization(), .AllowAnonymous(), [Authorize])
  • Использовать NoContent, вместо Ok, если нет содержания. NoContent избавит ответ от заголовка Content-Length: 0
  • Желательно явно указать привязку, например [FromBody]
  • CancellationToken
Вызов эндпоинта
  • Проверки токенов, если нужно для сервиса
  • Проверки данных (пустой GUID, пустые данные, валидность. Бизнес проверки должны быть в сервисе, ответственность эндпоинта это предоставить валидные, допустимые данные)
  • Вызов сервиса
  • Вывод ошибки
  • Обработка конфликтов параллельности
  • Эндпоинт не должен взаимодействовать с базой, только сервис
  • Без исключений (если выбросится исключение о невалидности, то глобальный обработчик поймает, а позже выяснится недотест)
  • Положительный ответ (Ok, Json)

Вызов сервиса
  • Приставка Async
  • Пересмотри метод, тут точно исключение, а не результат
  • Если в метод пришла пустая модель, то исключение, это была ответственность эндпоинта ArgumentNullException
  • Если в метод пришла невалидная модель или GUID - исключение, т.к это была ответственность эндпоинта InvalidOperationException
  • Бизнес проверки через результат (существует ли пользователь, занят ли Username)
    *последовательность проверок тоже важна
  • Также после манипуляций с сущностью, её также надо проверить прежде, чем записывать в базу - падаем с исключением
  • Исключения и результаты на английском через константы ErrorMessages
  • Если в сервисе вызывается другой сервис, необходимо его ошибку обработать тоже, если есть ошибка, то просто в вызывающем сервисе возвращаем эту ошибку
    Также помимо результата этого внутреннего сервиса, нужно обработать его конфликты параллельности через try catch, если у него они есть и если это обработка больше, чем генерация результата, если это простой результат, то пусть обработается в эндпоинте
  • Само действие сервиса (обновление данных, создание)
  • Возврат успешного результата
  • На всём пути запроса не должны повторяться бизнес проверки, не нужно проверять по два раза, кроме валидации.
    *Дабы избежать повторных проверок во вложенных сервисах в XML-документации пишутся все бизнес проверки сервиса.

  • AsNoTracking()
  • Избегать повторения бизнес проверок
  • Соблюдать принципы ООП, например, не все свойства должны быть открытыми для set'а (инкапсулировать)
  • Вытаскивать из базы только необходимые данные .Where().Select(x => new { x.Username, x.Firstname, x.Id }).FirstOrDefaultAsync()
  • Ленивая загрузка и тд. Разобраться нужно Документация
  • Использовать Benchmark
  • Кэш GET, HEAD запросов (OutputCache), и вообще по коду (HybridCache). И всё это с Redis
  • IEnumerable, IQueryable
  • Везде использовать DateTime.UtcNow вместо DateTime.Now, это касается не только производительности, но и правильности

Обработка конфликтов параллельности
  • Если конфликт параллельности можно обработать лишь уведомлением пользователя в эндпоинте, то обработка, ложится полностью на него, как глубоко бы не появился конфликт он будет перехвачен эндпоинтом.
    А если конфликт нужно обработать, как-то иначе, например полностью откатить изменения, то тогда можно смело обработать и вернуть результат в самом сервисе

    • Если кратко, то, если тупо сгенерить результат о конфликте — это эндпоинт
    Если, обработка конфликта сложнее — то сразу на месте
Логирование
  • Логировать редкие, не очень ожидаемые результаты или ошибки.
    Например, результат "аватарка не найдена", такое не должно случаться просто так. В приложении не может просто так пропасть аватарка пользователя
Значение проектов
  • Benchmarking - абсолютно разные бенчмарки по всему проекту

  • CRUD.DataAccess - слой доступа данных монолита
  • CRUD.Models - модели монолита
  • CRUD.Services - сервисы монолита (слой бизнес-логики)
  • CRUD.Utility - утилиты монолита
  • CRUD.Tests - тесты монолита
  • CRUD.WebApi - WebApi (по сути UI монолита, если по определению - конечные точки, которые дергает клиент)

  • CRUD.WebApplication - веб-сайт клиент этого API

  • Microservice.EmailSender - микросервис отправки электронных писем
  • Microservice.EmailSender.Tests - тесты микросервиса

  • CRUD.Infrastructure.S3 - S3Manager, вынесенный в отдельный nuget-пакет, т.к этот сервис используется и монолитом, и микросервисом
  • CRUD.Infrastructure.S3.Tests - тесты S3Manager'а

  • CRUD.Shared - общие файлы, используемые монолитом и микросервисом (отдельный nuget-пакет)
  • CRUD.Test.Shared - тестовые общие файлы, используемые всеми тестовыми проектами

    • nuget-пакеты нужны для версионирования, чтобы монолит и микросервис(ы) могли плавно перейти к актуальной версии (когда угодно, а не когда обновятся общие файлы).
